Abstract

This document describes the notation and semantics of an Abstract System for Computation, abbreviated as the name "ASC". ASC is an extended algebraic language which is useful for describing and reasoning about complex algorithms and programs.

Introduction

An Abstract System for Computation is a notion of a particular class of programming langauges, one of which is described in this document. This notion has several considerations which sets it apart from the common understanding of what a programming langauge is.

  • It is more appliccable to determining how the answer to a problem may be found, or if a problem can be answered, rather than determining the significance of an answer once it's found.
  • It allows problems to be represented as they appear in their domain, rather than requiring the parameters to be translated to a different domain.
  • It includes detailed modelling facilities for reasoning about algorithms in general and in abstract.

Foundation

ASC is based in a meta-language for describing geometrical objects, their properties, and their relations. There is such a close correspondence that the notation of ASC is largely the notation for the meta-language. Because of this, the meta-language is only described naturally. The actual notation for it is given in later sections regarding constructing the corresponding concepts in ASC.

This meta-language is used specifically for creating models of geometry which may exist. An object described in this language can be thought of as the "type" of the actual geometry which it models. It is important to note that it is possible and relatively simple to model geometry which does not and cannot exist.

Definition of Type

  • A type is some geometric formation represented as a collection of features.
  • A feature is some aspect of a geometric form which may not be real or even possible.
  • Each type has a countable but unbounded number of features beginning with none.
  • Types may have more than one identical feature.
  • Multiple types may have identical features to each other. A feature which is shared between types is a common feature.
  • There is a single type with no features. It is the Null type.
  • There is a single type with every existing feature. It is the Any type.

Relationships Between Types

  • Consider two types where every feature between them is common. These two types are equivalent.
  • Consider two types where no feature between them is common. These two types are disjoint.
  • Consider two types where at least one and not all features are common between them. These two types are alike.
  • Consider two types where every feature of one type is common, and at least one feature of the other is not common. The type with the least number of features is the sub-type. The type with the greatest number of features is the super-type.

Meta Features

There are several features which represent correspondences between types in the meta-language.

Unit
A type which has exactly one feature is a unit type. Unit types identify their feature.
Function
A function is a feature which represents a transformation from some features in a collection of types to some features in another collection of types.
Collection
A type which is the super-type to one or more types.
Superposition
A feature which is possibly any of a collection of features and must be identical to one.

Semantics

ASC provides a number of fundamental entity types which are composed into programs. They are,

  • Variables
  • Data
  • Types
  • Collections
  • Functions
  • Modules
  • Notations

Except for Notations, all entities may be used in computation.

Representing Programs

An ASC program is represented as a streaming sequence of characters. Notations are used to decide how the stream of characters corresponds to expressions.

Lexical Conventions

An ASC program is represented by a stream of characters. Default behavior is provided to analyze the stream and transform it into a stream of symbols and expressions.

Implementations of the ASC language are encouraged to support the superior superset of ASCII, the Unicode character encoding, which features tens of thousands more characters in multiple languages.

Character Classifications

Characters as they are used in ASC are classified according to whether they are visible or invisible. Visible characters are any characters which have a corresponding glyph. Invisible characters are any characters, including blank space, which have no corresponding glyph.

Classification for the ASCII Encoding

The character codes 33 to 126 are visible characters.

The character codes 0 to 32 and 127 are invisible characters.

Classification for Other Encodings

If alternate encodings are used in an implementation, the normal classification of visible and invisible characters must still be applied. Character codes which correspond to printed glyphs should be considered visible. Character codes which correspond to white space, separators, or other forms of character printing control should be considered invisible. An encoding may include special codes for formatting or modifying the appearance of visible characters. These character codes should be considered visible characters.

Symbols

Symbols are the basic element of ASC expressions and are composed of sequences of characters. Symbols may be featured in expressions as delimiting elements of an expression, variables, or numerical constants. The characters which are included in a symbol are decided by "grouping" rules.

Character Grouping Rules

  1. Visible characters shall be included as part of a symbol.
  2. Invisible characters shall delimit symbols such that one symbol ends before the delimiter and another begins after it.
  3. Two adjacent visible characters shall be included in the same symbol.
  4. Two adjacent delimiting characters shall be treated as a single delimiting character.
  5. A notation may define a specific sequence of characters which mark an expression. This sequence of characters will be treated as a delimiting character where it occurs.

Quoting Symbols

A symbol may be prefixed with a backquote, `, in which case, the symbol becomes "quoted". A quoted symbol is neither evaluated or consumed as a delimiter but appears as a unique symbolic value defined by the characters which compose it. A quoted symbol will always have the same value as the quotation of the same symbol, which means `s is equivalent to `s and not equivalent to `t. Additionally, the fifth character grouping rule is ignored. All visible characters following the initial backquote will be included in the quoted symbol. Invisible characters occurring between the initial backquote and the first visible character will be discarded.

Quotation on symbols may be removed by prefixing the quoted symbol with the "unquote" prefix, ~` in which case the symbol will be evaluated. This prefix can be used directly on a quoted symbol as in ~``symbol or with a quoted symbol that has been named such as ~`(x = `symbol ). In both these examples, the symbol, symbol will be evaluated.

Constants

If a symbol is composed entirely of decimal digits (one of the characters 0123456789), then it is a numerical constant.

Constants are always interpreted using the decimal (base-10) number system. The lowest place value begins with the last, rightmost decimal digit and increases towards the first. For Naturals and Integers, the lowest possible place value is 100. For Rationals, the lowest possible place value is 10-n where n is the number of digits trailing to the right of the decimal dot.

Arithmetic operations between two constants are guaranteed to yeild another constant.

Naturals

Whole, positive numbers are represented simply by sequences of decimal digits such as 1 or 1234987605. Leading zeros are permitted such as 0000001.

Integers

Integers allow the inclusion of a sign, either positive + or negative -, as the first character in the number. This allows for positive and negative whole values such as +100 and -42.

Rationals

Rational values allow for the inclusion of either a decimal dot, ., or a fraction sign, / as long as there is a decimal digit directly adjacent on either side of the decimal or fraction symbol. Example rational values are -1/3 and +0.3333333. It should be noted as seen in the example that rational numbers are capable of expressing any infinite sequence which can be written as a proper fraction.

Environment

Entities which are constructed as part of a program exist within a discrete environment. This environment serves to abstract over whatever actual resources may be present to perform computation. Each entity which exists in the environment has an established location, duration, and size.

By default, all entities are accessible to each other within the environment. Artificial boundaries may be introduced to limit which entities may access others by establishing contexts. Contexts are implicitly established and dissolved through the evaluation of a number of specific entities.

Location

There is a finite, countable number of locations within the environment, constrained by the capabilities of the underlying computing resources. When a new entity is introduced to the environment, it is stored at a location. This location will never change for the duration that the entity exists in the environment.

Locations appear in the environment as entities themselves and can be computed with as values. Locations can be derived from an entity's placement within the environment. Data can also be accessed from locations.

Locations as values for computation are not ordered and do not have the property of magnitude. Every location available in the environment is a unique value which has no relationship with any other locations. The only comparison that can be made is equality: whether two location values represent the same location, or whether those values are different.

Contexts

By default, every entity may access every other entity in the environment. Contexts establish a means of limiting how entities may access each other. Entities may either reside in or out of the context. Entities within the context may access each other, but entities outside the context cannot access any entity inside the context.

Notations

Notations are functions* with extended facilities to describe forms of expressions. Instead of relying on function application, notations are applied to parameters by writing the expression that they describe. There are two main extensions which notations provide, contexts and patterns. Notation definitions are made distinct from function definitions by prefixing them with a semicolon.

A single notation will only correspond to one form of expression. An expression may have any number of parameters, keywords, and delimiting characters. No part of an expression may be optional. An exception to this is variadic parameters.

Notation Semantics

Engaging A Notation

Notations do not have an explicit application expression like functions. Instead, they are applied when an expression occurs which matches the pattern for a notation. When an expression matches a pattern defined by a notation, that notation is engaged. After a notation is engaged there are additional steps which are performed before normal function evaluation occurrs.

Capturing Expressions

If a notation has been engaged, and it accepts parameters, then an expression will be captured for each parameter. The expressions captured must be in positions corresponding to where parameters were given in the notation pattern. For example, given the following expression and notation,

Example Expression
PREFIX x + 1
Example Notation
; (`PREFIX n) = n + 1

the expression, x + 1 will be captured when the notation is engaged. When the notation is evaluated, the parameter n will be the result of evaluating x + 1.

The expressions which are captured form a context. When these expressions are evaluated, additional notations may engage which depend on that context being established.

Contexts

A notation definition may optionally begin with a context. Contexts provide a mechanism for notations to conditionally require additional notations to be engaged first before they can engage. A context takes the form of a list of symbols occurring after the semicolon prefix and before the pattern definition. The last symbol corresponds to the name of the context for the notation being defined. If the notation does not require a named context, then the last symbol in the list must be a period dot. If the list consists of a single symbol with no period following, then that symbol names the context of the notation being defined.

The context list is treated as a hierarchy. This means that the each context in the list must be present before the one following it can be engaged. All contexts listed must be engaged before the notation being defined can engage. For example, given the three following notations which contextually depend on each other,

Notations With Context Dependencies
; a (`a x) = x
; a b (`b x) = x
; a b . (`c) = 1

The following expression would be necessary to engage the last notation with no named context.

Expression Satisfying Context Dependencies
a b c

Names and Accessiblity

Notations are a meta-language facility. This means that notations are not accessible as entities and their contexts aren't variable names.

Example Contexts

An empty context list, ; represents a notation context which does not rely on any contexts and does not name a context of its own. The list, ; m represents a notation context which does not rely on any contexts, but names it's own context "m".

Given the context list, ; a b c ., a notation with context "a" would have to engage. Then within that, a notation with context "b" would have to engage. Finally, finally a third notation with context "c" would have to engage before the pattern for this hypothetical notation could be compared against the stream.

Patterns

A pattern is an extension of the function parameter definition. Patterns define the collection of possible sequences of symbols which can engage the notation.

Patterns extend the normal function parameter list with mechanisms for declaring delimiting characters, symbols, repeating elements, and explicit evaluation.

Strings

Strings correspond to the fifth character grouping rule where a series of characters may be treated as a delimiter. A string uses the default string notation* to define one or more characters in a sequence which serves to identify an expression.

When a notation engages, any strings featured in its pattern must be discarded from the captured context.

Symbols

Symbols are defined in a pattern using symbol quotation*. Symbols are used in patterns in the same way as strings, to identify a specific expression. However, symbols follow the normal character grouping rules*. In order for a symbol in a pattern to match a stream, that symbol must occurr explicitly as a symbol in the stream.

Like strings, symbols must also be discarded from the captured context when a notation engages.

Parameters

A pattern may include function parameters*. These provide "openings" in the form of an expression where further expressions may be nested.

Function parameters behave identically in both functions* and notation patterns. Expressions may be nested within the notation at the point where a parameter. Parameters in notations must be supplied entities through expressions being nested in positions corresponding to a parameter's placemnt within a pattern.

Semantics

Errors

Errors may be encountered during the translation of a program into an interpretable format. They may be caused by various undesireable conditions. In the event of an error, translation should abort. The implemntation should also communicate the nature of the error and its position within the stream. The list below describes the various classes of errors which may occur during translation.

Classes of Translation Error
Lexical Errors
A lexical error occurs when a character sequence is encountered in the stream for which there is no corresponding notation.
Semantic Errors
A semantic error occurs when an expression is mal-formed such as notations being engaged in an unexpected manner. An example of a semantic error is two addition operators being written adjacent to each other.
Example of a Semantic Error
+ +
Type Errors
A type error is similar in nature to a semantic error, except it occurs specifically when notations are composed in a semantically valid manner, but the resulting expression is not sound according to the mechanics of the type system. An example is a numerical addition performed with entities which are not numbers.
Example of a Type Error
`x + `x
