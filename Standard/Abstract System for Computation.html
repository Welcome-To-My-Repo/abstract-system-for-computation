Overview

This document describes the properties of an Abstract System for Computation (ASC), a general purpose tool for computing. ASC consists of two interdependent components, an algorithmic language and an environment for computing programs described in that language.

Scope

This document specifies...

  • the syntactic and semantic properties of the ASC language.
  • the capabilities and limitations of the ASC environment.
  • all assumptions made about the world outside the environment.
  • any data which should be provided by default
  • the methods for interpreting the ASC language

This document does not specify...

  • requirements for anything outside the environment.
  • methods for translating the ASC language to another language
  • the storage format of the environment and data in it

Features

ASC provides a session-based interface with one or more bidirectional streams. These streams convey character data representing the ASC language. Data can be entered into the environment for computation through these streams. The environment can send data back out across the stream using a pre-defined display function.

The environment holds all data in an image. An image is an archival format which can be exported from and imported to a session. This allows data to persist between multiple sessions.

ASC represents programs as data. The ASC language as it is written is translated to a structured data format before it is computed. The process of computation involves inspecting the program data structure for pre-defined substructural patterns and performing transformations.

Environment

The environment facilitates the process of computation. It provides some basic capabilities and abstractions such that all ASC programs recieve the same view of the external environment.

The Compute Function

The Compute function accepts a program model. It will iteratively reduce the structure of the model until a single constant is left. This process is not required to halt.

The Compute function will also accept a String instead of a program model. It will apply the Read function to that string and then apply itself to the program model resulting from that application.

Compute performs a reduction in the following steps...

  1. Find the first (leftmost) expression which does not contain a nested expression.
  2. Interact with the Context using that expression.
  3. Substitute that expression with the Reaction.

Choosing an Expression

A simple algorithm for determining which expression should be used to interact with the Context is to chose the expression which can be reached with the smallest sum of indicies. Given that the program model consists of nested Collections, every nested expression can be reached by index subscripting. It follows that the first expression which does not contain another nested expression should be reachable by the smallest index for each subscript operation performed. Thus finding the minimal sum of index values to reach an expression will identify the first expression.

The Read Function

The Read function accepts a String containing a written program. It will build a program model which must be equivalent to the written program.

Deciding Equivalence

The equivalence of a program model to a written program is decided by observing if the Contexts resulting from computing both forms of the program are equivalent. Thus equivalence between two programs can be expressed as the following program,
(Pm), A <- %, (Pw), B <- %, A = B
where Pm corresponds to a program model and Pw corresponds to a written program and computing this program results in a boolean value signifying the equivalence or non-equivalence of the two programs.

The Display Function

Display accepts any object and will emit a String containing a written program that results in that object when computed. Display will always produce a well-formed written program such that
x : *, Compute Read Display x = x

Data

The environment assumes at least two capabilities: the ability to hold data temporarily while a transformation is being applied and the ability to store data in between transforms. Two formats are required, one for performing transformations on data and one for storing data. The Word is the format for transforming data. The Byte is the format for storing data.

If multiple storage or transform formats are available to the environment, then they may be provided in ASC as data formats in addition to Words and Bytes.

The Byte

The Byte corresponds to a single unit of storage. All entities in the Environment may be stored using one or more bytes. A Byte format is assumed to be capable of representing some number of whole, positive values. A Byte must be convertable to a Word value.

In the case of multiple viable formats being available to the Environment, the Byte must correspond to the smallest uniquely identifiable storage unit available to the Environment. The ability to uniquely identify a single unit may include indirection.

Size

The number of Bytes required to completely store an entity is referred to as the "size" of an entity. The size of an entity is considered a constant value and does not require computation. A program may query the size of any entity.

Location

The Environment assumes that the available storage resources collectively provide a countable number of Bytes. Each byte may then be assigned a unique whole, positive number. The Environment assumes this numbering is arbitrary and does not correspond to the physical position of bytes in a storage medium. The unique number identifying a Byte is considered a constant value and does not require computation. A program may query the identifying number of a Byte.

The Word

The Word corresponds to the the implementation specific format used for transforming data. The Word format must allow the basic arithmetic operations, addition, subtraction, multiplication, division yielding a quotient, and division yielding a remainder. These operations shall follow the natural behavior of arithmetic over the set of Integers as far as the Word format is capable of representing Integer values. If any of these operations are available to the environment as part of the implementation specific Word format, then the environment is not required to simulate those operations and may use the ones supplied with the format.

In the case of multiple viable formats being available to the Environment, the Word must correspond to the format that offers the most complete set of arithmetic operations. Additionally, the format which is most natural to the computer must be used. That is to say a format which is preferable for the Environment may not be the format most conducive to efficient computation. This may disqualify some formats which would otherwise be judged suitable. The ultimate choice of format is completely implementation dependent and is purposefully left vague.
