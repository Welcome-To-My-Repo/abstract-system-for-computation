Abstract

This document describes the notation and semantics of an Abstract System for Computation, abbreviated as the name "ASC". ASC is an extended algebraic language which is useful for describing and reasoning about complex algorithms and programs.

Introduction

An Abstract System for Computation is a notion of a particular class of programming langauges, one of which is described in this document. This notion has several considerations which sets it apart from the common understanding of what a programming langauge is.

  • It is more appliccable to determining how the answer to a problem may be found, or if a problem can be answered, rather than determining the significance of an answer once it's found.
  • It allows problems to be represented as they appear in their domain, rather than requiring the parameters to be translated to a different domain.
  • It includes detailed modelling facilities for reasoning about algorithms in general and in abstract.

Foundation

ASC is based in a meta-language for describing geometrical objects, their properties, and their relations. There is such a close correspondence that the notation of ASC is largely the notation for the meta-language. Because of this, the meta-language is only described naturally. The actual notation for it is given in later sections regarding constructing the corresponding concepts in ASC.

This meta-language is used specifically for creating models of geometry which may exist. An object described in this language can be thought of as the "type" of the actual geometry which it models. It is important to note that it is possible and relatively simple to model geometry which does not and cannot exist.

Definition of Type

  • A type is some geometric formation represented as a collection of features.
  • A feature is some aspect of a geometric form which may not be real or even possible.
  • Each type has a countable but unbounded number of features beginning with none.
  • Types may have more than one identical feature.
  • Multiple types may have identical features to each other. A feature which is shared between types is a common feature.
  • There is a single type with no features. It is the Null type.
  • There is a single type with every existing feature. It is the Any type.

Relationships Between Types

  • Consider two types where every feature between them is common. These two types are equivalent.
  • Consider two types where no feature between them is common. These two types are disjoint.
  • Consider two types where at least one and not all features are common between them. These two types are alike.
  • Consider two types where every feature of one type is common, and at least one feature of the other is not common. The type with the least number of features is the sub-type. The type with the greatest number of features is the super-type.

Type Structures

These structures represent various correspondences that appear as types, but express properties that cannot be directly represented by types themselves.

  • A unit is a type which has exactly one feature and consequently serves to identify that feature.
  • A function is a type which represents a transformation from one collection of types to another.
  • A collection is a type which has a common feature with every feature of one or more types.
  • A superposition is a type which is possibly equivalent to any one of a collection of types and cannot be disjoint to every type in the collection.

Notations

Notations are the means through which written expressions in the ASC language are mapped to the semantics of ASC algorithms. They achieve this in two stages, by transforming a stream of characters into a stream of symbols and transforming a stream of symbols into some expressions in a program.

Implementations of the ASC language are encouraged to support the superior superset of ASCII, the Unicode character encoding, which features tens of thousands more characters in multiple languages.

Character Classifications

Characters as they are used in ASC are classified according to whether they are visible or invisible. Visible characters are any characters which have a corresponding glyph. Invisible characters are any characters, including blank space, which have no corresponding glyph.

Classification for the ASCII Encoding

The character codes 33 to 126 are visible characters.

The character codes 0 to 32 and 127 are invisible characters.

Classification for Other Encodings

If alternate encodings are used in an implementation, the normal classification of visible and invisible characters must still be applied. Character codes which correspond to printed glyphs should be considered visible. Character codes which correspond to white space, separators, or other forms of character printing control should be considered invisible. An encoding may include special codes for formatting or modifying the appearance of visible characters. These character codes should be considered visible characters.

Symbols

Symbols are the basic element of ASC expressions which are composed of sequences of characters. They have a dual role of being evaluated as names of entities or being consumed as delimiting marks for individual notations. Which characters are included in symbols, which ones are excluded, and where individual symbols begin and end in a stream of characters are decided by a set of character "grouping" rules. These rules are used to decide how a symbol is formed from a stream of characters.

Character Grouping Rules

  1. Visible characters shall be included as part of a symbol.
  2. Invisible characters shall delimit symbols such that one symbol ends before the delimiter and another begins after it.
  3. Two adjacent visible characters shall be included in the same symbol.
  4. Two adjacent delimiting characters shall be treated as a single delimiting character.
  5. A notation may define a specific sequence of characters which mark an expression. This sequence of characters will be treated as a delimiting character where it occurs.

Quoting Symbols

A symbol may be prefixed with a backquote, `, in which case, the symbol becomes "quoted". A quoted symbol is neither evaluated or consumed as a delimiter but appears as a unique symbolic value defined by the characters which compose it. A quoted symbol will always have the same value as the quotation of the same symbol, which means `s is equivalent to `s and not equivalent to `t. Additionally, the fifth character grouping rule is ignored. All visible characters following the initial backquote will be included in the quoted symbol. Invisible characters occurring between the initial backquote and the first visible character will be discarded.

Quotation on symbols may be removed by prefixing the quoted symbol with the "unquote" prefix, ~` in which case the symbol will be evaluated. This prefix can be used directly on a quoted symbol as in ~``symbol or with a quoted symbol that has been named such as ~`(x = `symbol ). In both these examples, the symbol, symbol will be evaluated.

Defining Notations

Notations are described in three distinct components, the context, the pattern, and the definition. As a stream of characters is consumed, the patterns for all accessible notations are compared against the stream. If a pattern matches against the stream, the corresponding notation "engages". When a notation "engages", it performs two things, it first consumes the part of the stream which matches it pattern which becomes the context for that notation. Secondly, the corresponding definition for that notation is evaluated. After the definition has been evaluated, the consumed portion of the stream is discarded.

Contexts

Contexts provide a mechanism for notations to conditionally depend on a separate notation being engaged before their pattern is compared against a stream. When a notation engages and the part of the stream matching its pattern is consumed, the patterns notations which depend on the context of the engaged notation are compared against the portion of the stream being consumed.

Contexts are written beginning with a semicolon prefix followed by a series of symbols which name contexts. The series is hierarchical meaning that each context must exist in the one it was listed after. The last context given must be the name of the context created when the notation being defined engages. If the context isn't named, then the place of the last name is filled with a period dot.
