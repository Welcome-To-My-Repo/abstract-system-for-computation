An Abstract System for Computation

Draft Revision 0.0


Abstract

This document describes the notation and semantics of an Abstract System for Computation, abbreviated as the name "ASC". It provides advanced features which are useful for expressing and reasoning about complex algorithms and computer programs.

Introduction

Properties of an Abstract System for Computation

An Abstract System for Computation is a concept which is both specific to this document and defines the philosophy involved with the creation of ASC. The key consideration is that an abstract system for computation is a higher abstract concept which the programming language, ASC, is only intended to represent. There may very well be other, alternative systems for programming or mathematical reasoning which can also be described as an abstract system for computation, but they are not defined here. In fact, the title of this document uses "An" instead of "The" to make this point explicit. The programming language described in the following chapters is being proposed as one of many possible manifestations of an abstract system for computation.

An abstract system for computation is mainly a system for mathematical and logical reasoning. The manner in which it represents structured information and algorithms is based in mathematics rather than pieced together nominally. Every component is formalized or founded in an external theory.

An abstract system for computation is different from mathematics because it addresses different concerns. It is a lower level of abstraction than most mathematics. Classical mathematics is concerned with the correspondence between problems and answers. By contrast, abstract systems for computation are concerned with how an answer can be obtained and if solving a problem is possible.

Language Overview

ASC consists of a number of fundamental entities which are composed to form programs. Programs are represented textually through written expressions which construct entities and apply functions. The following descriptive list outlines the main classes of entities in no particular order.

Notations
Notations provide a meta-language utility for creating new forms of syntax and mapping them to some program semantics.
Functions
Functions represent computations to be performed. Every expression is some form of function definition or application.
Data
Data provides a correspondence between values represented at the program level and values as they can be represented in the environment where ASC programs are executed.
Types
Types provide a means to create generic models representing a specific class of program. If it is possible to specify a program using types in a sound way, then executing the actual program is guaranteed to function according to the model.
Modules
Modules separate a cohesive program into discrete components which interact using structured interfaces.
Sequences
Sequences allow data to be organized into various kinds of structures. More generally, sequences allow multiple individual pieces of data to be indexed and named within a single abstract entity which appears as data itself.

Syntax

ASC programs are represented by a sequence of characters which is converted into a corresponding sequence of symbols using a set of grouping rules. Symbols are then mapped to program semantics using a set of notation rules. Notations may be defined as part of the program itself and can affect how symbols and expressions are identified.

Symbols

Symbols are the most basic program-level entity used to build programs. They consist of sequences of visible characters delimited by blank, unprintable, and control characters.

Character Grouping Rules

  1. Visible characters shall be included as symbols. For the ASCII character encoding, visible characters include the codes 33 to 126.
  2. Blank, unprintable, or invisible characters shall delimit symbols. For the ASCII character encoding delimiting characters include the codes 0 to 32 and 127.
  3. Two adjacent visible characters shall be included in the same symbol.
  4. Two adjacent delimiting characters shall be treated as a single delimiting element.
  5. A specific sequence of characters may be defined using Notations which is treated as a complete symbol and will delimit any visible characters adjacent to itself.

Examples of Symbols

The character sequence
abc def ghi
corresponds to the three symbols abc, def, and ghi.

The character sequence
a&* &9)(li afj i3#afs
corresponds to the four symbols a&*, &9)(li, afj, and i3#afs.

Note that no distinction is made between symbolic and script characters. This allows for names which may include symbols.

Regarding Alternate Encodings

If alternate character encodings are used, the rules for grouping characters into symbols should be such that any visible characters compose symbols and any non-visible characters delimit symbols. It may be necessary to transparently ignore certain characters such as formatting codes which are technically invisible, but change the textual appearance of a visible character. Alternatively, additional rules may be necessary to decide if a character is delimiting or used in symbols. The task of interpreting and applying these rules to alternative character encodings is left to the implementation.

Notations

ASC is self-describing through Notations. A notation defines some syntax and the semantics that are engaged when the syntax occurs in a program. Notation declarations are prefixed with the semicolon, ;, and are composed of three parts, the context, the pattern, and the definition.

Notation Semantics

Notations fulfill the dual role of grouping sequences of characters into sequences of symbols and mapping those symbols to expressions. Thus notations follow the character grouping rules* by default and can override them depending on the pattern defined. When a pattern for which a notation has been defined occurs in a sequence of characters, that notation is "engaged". The process of a notation being engaged is identical to a function being called in that the definition component of the engaged notation will be evaluated.

Contexts

When a pattern occurs and a notation is engaged, that sequence of characters corresponding to the pattern is said to be "captured". This captured sequence of characters is a context. Notations can be defined in such a way that they will only engage if their corresponding pattern occurs within a previously captured sequence.

A notation context is defined as a list of context names which must be engaged before the notation being defined can be engaged. The list is interpreted in hierarchical order such that the first listed context following the semicolon prefix must be engaged before following contexts may apply.

The last context in the list of context names is interpreted as the name of the context for the notation being defined. If a notation does not require a named context, the last name must be a period. If a notation is given a context name, it must be a unique name which is not already used to name any other context accessible to the notation being defined.

Patterns

A pattern is composed of three elements: characters, symbols, and parameters. These three elements are used to decide how a pattern may appear in a sequence of characters.

Characters and symbols act as terminal components in a pattern. They delimit or otherwise mark the structure of a pattern and are never evaluated.

Characters

A pattern may specify one or more characters in a sequence which are treated as their own symbol. Character sequences in a pattern are written as a string. The string may contain any characters including invisible ones. Normal grouping rules are still applied to the remaining parts of the character sequence. The string itself is not included in the remaining sequence and is treated as a single delimiting character.

Symbols

A pattern may include symbols for which normal character grouping rules apply. Symbols which are specified as part of a pattern can no longer be used to make entities in expressions.

Formalization of Types

ASC utilizes a system of types which is founded in the description of geometric objects and their relations. In this way, types can be extended to reflect any theory of geometry. Judgements about type can then be enhanced by building on pre-existing relationships established in other fields of mathematics.

Additionally, this system of types is neither complete nor consistent. In fact, it explicitly allows for contradictory or absent information. The intent is that consistency and completeness are sacrificed for greater expressiveness. Mechanisms for handling illogical information can be used to reason about programs or environments which present illogical states and behavior. This kind of system acknowledges that a judgement in real life may depend on multiple gradual stages of reasoning or even missing information.

Definition of Type

  1. A type is some abstract geometric form with features.
  2. A feature is some consideration about a geometric form such as dimension, size, shape, or any arbitrary term.
  3. Every type has a countable number of features from none to infinite.
  4. Multiple types may have the same feature. A feature shared between two types is called a common feature.
  5. There is a single type which has no features and it is the Null type.
  6. There is a single type which has every possible feature and it is the Any type.

Type Relationships

  1. Consider two types where every feature between them is common. These types are equivalent.
  2. Consider two types where every feature between them is not common. These types are disjoint.
  3. Consider two types where at least one feature is common. These types are alike.

Type Structures

  1. A unit is a type which has a single feature and consequently serves to identify that feature.
  2. A function is a type which represents a transformation between two collections of types.
  3. A composition is a type which has a common feature for every feature present in a collection of types.
  4. A superposition is a type which is possibly equivalent to one of a collection of types.

These structures represent various correspondences which appear as types, but express properties that are external to the definition of types as it is given here.
